# سند — دليل المشروع (CLAUDE.md)

> ملف توجيه للمطوّر ولوكلاء Claude Code العاملين على هذا المستودع. يشرح الهدف والبنية
> ونظام الأدوار الثلاثة وآلية التزامن عبر الشبكة المحلية، مع دليل التشغيل واختبار التزامن.

## 1. هدف المشروع

**سند** تطبيق سطح مكتب (Electron) لإدارة ومحاسبة الأعمال الصغيرة (نقطة بيع + محاسبة)،
ثلاثي اللغة (عربي/فرنسي/إنجليزي، RTL)، مدعوم بذكاء اصطناعي (Groq) وبوت تلجرام.

النسخة الحالية تدعم **تشغيل عدة أجهزة على نفس الشبكة المحلية** بتزامن لحظي: جهاز **المدير**
يعمل كخادم رئيسي (host) يملك النسخة المرجعية من البيانات، وأجهزة **المحاسب** و**الكاشير**
تتصل به (clients) فتتزامن العمليات والفواتير والمخزون بينها فوراً.

**قرار معماري (مثبَّت مع المالك):** نسخة واحدة من التطبيق تُثبَّت على كل الأجهزة —
الدور (مدير/محاسب/كاشير) يُختار **عند تسجيل الدخول**، ودور الشبكة (مضيف/متصل) يُضبط
**محلياً في كل جهاز** من الإعدادات. لا توجد مثبّتات منفصلة لكل دور.

## 2. البنية العامة

| الملف | الدور |
|------|------|
| `main.js` | عملية Electron الرئيسية: تخزين JSON، الذكاء الاصطناعي (Groq)، بوت تلجرام، PDF، النسخ الاحتياطي، الترخيص، **طبقة التزامن عبر الشبكة (LAN)**، ومعالِجات IPC. |
| `preload.js` | جسر آمن (`contextBridge`) يكشف دوال IPC للواجهة تحت `window.sened`. |
| `renderer/index.html` | هيكل الواجهة وكل الصفحات والنوافذ. |
| `renderer/app.js` | منطق الواجهة: العرض، النماذج، الصلاحيات، تسجيل الدخول، **واجهة إعداد الشبكة والتحديث اللحظي**. |
| `renderer/i18n.js` | قواميس الترجمة الثلاثية (`I18N.ar/fr/en`). |
| `renderer/styles.css` | التنسيق (سمة داكنة/فاتحة، RTL). |
| `assets/` | الشعار والأيقونات. |
| `builtin-ai.json` | إعداد مدمج (gitignored): مفتاح Groq المشترك + كود تفعيل المطوّر. لا يُرفع للمستودع. |

**التخزين (منذ v1.1.0): قاعدة SQLite** في `‎userData/sened.db` (عبر better-sqlite3، WAL):
جدول `records(collection,id,updatedAt,deleted,data)` — كل فاتورة/منتج/زبون صفٌّ مستقل،
والإعدادات والعدّادات صفوف `_kv`. الحذف شاهدةٌ (`deleted=1`) كي ينتشر للأجهزة. الترتيب
الأصلي محفوظ بـ rowid (سجل التدقيق والسلة يُرتَّبان أحدثاً-أولاً عند التجميع في `getDoc`).
الواجهة وبقية main.js تتعامل بمستند كامل كما كانت (`loadData/saveData/getDoc`) — طبقة
التخزين تحسب الفروقات سجلاً سجلاً (`diffDocToChanges`) وتطبّقها (`applyChangesLocal`).
ملف `sened-data.json` القديم يُرحَّل تلقائياً عند أول تشغيل ثم يُعاد تسميته `.pre-sqlite`.
النسخ الاحتياطية تبقى **بصيغة JSON الكاملة** (تُصدَّر من القاعدة) في `‎…/backups/`
(آخر 30 نسخة، كل ساعة + عند التشغيل) — قابلة للاستعادة في أي إصدار.
متغيّر بيئة `SENED_USERDATA` يعزل بيانات نسخة ثانية للاختبار على نفس الجهاز.

## 3. الأدوار الثلاثة (مدير / محاسب / كاشير)

الأدوار تُطبَّق عبر **تسجيل الدخول** ونظام صلاحيات `PERMS` في `renderer/app.js`:

```js
const PERMS = {
  manager:    null, // كل الصفحات
  accountant: ['dashboard','invoices','quotations','purchases','products','customers','suppliers','debts','wallets','expenses','reports','assistant'],
  cashier:    ['dashboard','invoices','quotations','products','customers']
};
```

- **المدير (manager):** كل الوحدات — إدارة المستخدمين والصلاحيات، التقارير والإحصائيات،
  المساهمون، الموظفون، رؤية صافي الربح والأرصدة.
- **المحاسب (accountant):** الفواتير، المشتريات، الديون، المحافظ، المصاريف، التقارير المالية —
  دون إدارة الموظفين/المساهمين ولا إعدادات النظام الحساسة.
- **الكاشير (cashier):** واجهة بسيطة سريعة — إصدار الفواتير وعروض الأسعار وإدارة المنتجات والعملاء.
  لا يرى صافي الربح ولا الديون ولا أرصدة المحافظ.

**رقم حساب الزبون (Numéro de compte):** كل زبون يحمل `accountNo` متسلسلاً بصيغة `C-0001`
يتولّد تلقائياً في `saveCustomer` (`nextCustomerAccountNo` في `app.js`)، ويظهر في جدول العملاء
ونافذة الزبون وقائمة اختيار الزبون في الفواتير والبحث (Ctrl+K). الزبائن القدامى يُرقَّمون عند
أول تعديل عليهم.

حساب المدير في `settings.auth`؛ حسابات المحاسب/الكاشير في `employees[]` (لكل موظف
`username` + `passwordHash` مملّح + `accessRole`). التحقق في `ipcMain.handle('auth:verify')`.

**مهم:** الصلاحيات تُفرض عند **نقطة النقر الفعلية** على عنصر التنقل وفي لوحة الأوامر (Ctrl+K)،
وليس بإخفاء CSS فقط. أي صفحة/إجراء جديد حسّاس يجب إدراجه في `PERMS` وفلترته في لوحة الأوامر.

### قفل دور الجهاز (Device Role Lock)
لتحقيق «تطبيق واحد، كل جهاز مقفول على دوره»: بعد **أول دخول بدور تشغيلي** (محاسب/كاشير)
يُربط الجهاز بذلك الدور فلا يقبل الدخول بدور مختلف، ولا يظهر إلا واجهة ذلك الدور، ويُوجَّه
مباشرةً لصفحته المناسبة (`LANDING` في `app.js`: الكاشير يبدأ من الفواتير، الباقون من لوحة التحكم).

- **التخزين محلي لكل جهاز** في `‎%APPDATA%/sened/device-role.json` = `{ role, boundAt }` —
  **خارج البيانات المزامَنة** تماماً مثل `lan-config.json` (وإلا لانتشر قفل جهاز واحد للبقية).
  الدوال في `main.js`: `readDeviceRole/writeDeviceRole`؛ IPC: `deviceRole:get/set/clear`؛
  الجسر في `preload.js`: `deviceRoleGet/Set/Clear`.
- **الربط** يحدث في `doLogin` عند أول دخول لدور غير المدير. **المدير يتجاوز القفل دائماً**
  (يبقى متنقلاً حراً وهو المخرج لإعادة الضبط) — دخول المدير لا يربط الجهاز.
- **إعادة التعيين/فكّ القفل**: للمدير فقط من الإعدادات (لوحة `#deviceRolePanel`، دوال
  `saveDeviceRole/clearDeviceRole` المحميّة بـ `currentRole==='manager'`). صفحة الإعدادات
  أصلاً خارج صلاحيات المحاسب/الكاشير فلا يصلها غير المدير.
- على جهاز بلا شاشة دخول (auth معطّل) لكنه مربوط بدور، يلتزم `bootAfterActivation` بذلك الدور.

## 4. آلية التزامن عبر الشبكة المحلية (LAN Sync)

### النموذج
جهاز واحد **مضيف (host)** — عادةً المدير — يشغّل خادم HTTP على المنفذ **3050**،
ويملك النسخة المرجعية من `sened-data.json`. بقية الأجهزة **عملاء (clients)**:
1. عند الفتح يسحبون الحالة الكاملة من المضيف (`GET /sync`).
2. يفتحون مجرى أحداث لحظياً (`GET /events` عبر **SSE**) فيستقبلون كل تحديث فور حدوثه.
3. عند أي تعديل محلي يرسلونه للمضيف (`POST /sync`)، فيخزّنه المضيف ويرفع رقم المراجعة
   ويبثّه لكل الأجهزة المتصلة.

### إعداد الشبكة محلي لكل جهاز
يُخزَّن في `‎%APPDATA%/sened/lan-config.json` — **وليس** ضمن بيانات التطبيق المزامَنة
(وإلا لانتشر عنوان/دور جهاز واحد لبقية الأجهزة). الشكل:
```json
{ "role": "off" | "host" | "client", "hostIp": "192.168.1.10", "port": 3050, "token": "" }
```

**رمز الحماية المشترك (اختياري):** إن ضُبط `token` (نفس القيمة على كل الأجهزة)، يفرضه المضيف
عبر middleware في `startHostServer` على كل الطلبات (ترويسة `x-sened-token`)؛ العميل يرسله في
`postToHost`/`connectClientStream`/`pullFromHost`، و`lan:test` يتحقق منه (401 = رمز خاطئ).
فارغ = معطَّل (توافق رجعي). يبقى محلياً في `lan-config.json` ولا يدخل البيانات المزامَنة.

### الاكتشاف التلقائي وكود الاقتران واسم الجهاز (جديد)
- **اكتشاف تلقائي (UDP):** المضيف يبثّ `{app:'sened', name, ip, port}` كل ثانيتين على منفذ
  **3051** (بثّ عام). العميل يضغط «البحث عن الخادم تلقائياً» فيستمع 3 ثوانٍ (`lan:discover` →
  `discoverHosts`) ويعرض الخوادم بأسمائها — اختيار بضغطة واحدة بلا إدخال IP يدوي.
  الدوال في `main.js`: `startDiscoveryBeacon/stopDiscoveryBeacon/discoverHosts` (تعمل مع
  الخادم وتتوقف معه).
- **كود الاقتران:** هو نفسه `token` — يتولّد تلقائياً (6 أرقام) على المضيف عند الحفظ إن كان
  فارغاً، ويظهر بخط كبير في لوحة المضيف؛ العميل يدخله عند الاتصال. الحفظ على المضيف يحافظ
  على الكود الحالي ولا يولّد كوداً جديداً.
- **اسم الجهاز (`deviceName`):** محلي في `lan-config.json`، يُبثّ في الاكتشاف، ويُرسل في
  ترويسة `x-sened-device` مع كل طلب، فيعرض المضيف أسماء الأجهزة المتصلة (`lan:status` →
  `clientNames`)، ويُلحق باسم المستخدم في سجل النشاط (`logAudit`: «user @ device»).
- **مؤشر التزامن:** شارة دائمة في الشريط الجانبي (`#syncBadge`) — أخضر: الخادم يعمل/متزامن،
  أحمر: انقطع الاتصال (يعمل محلياً ويتزامن عند العودة). تحديث كل 5 ثوانٍ + عند كل حدث مزامنة.

### نقاط النهاية (على المضيف)
| المسار | الوظيفة |
|-------|---------|
| `GET /ping` | فحص اتصال سريع؛ يعيد `{ok, rev, clients}`. |
| `GET /sync` | الحالة الكاملة الحالية (تحميل أولي للعميل). |
| `POST /sync` | استقبال تعديل من عميل: تخزين + رفع `_rev` + بثّ للجميع. |
| `GET /events` | مجرى SSE؛ يبقى مفتوحاً ويستقبل أحداث `update`. |

(بثّ UDP على المنفذ 3051 للاكتشاف التلقائي — خارج HTTP.)

### الدوال الرئيسية في `main.js`
`readLanConfig/writeLanConfig`, `applyLanMode` (يقرأ الدور ويشغّل المناسب)،
`startHostServer/stopHostServer`, `broadcastToClients`, `connectClientStream`
(SSE مع إعادة اتصال تلقائية كل 3 ثوانٍ)، `pullFromHost`, `postToHost`, `currentRev`.
معالِجات IPC: `lan:getConfig`, `lan:setConfig`, `lan:status`, `lan:test`.
التحديث اللحظي في الواجهة عبر الحدث `sync:updated` ← `onRemoteSync()` في `app.js`
(يعيد تحميل البيانات ويرسم، مع تجنّب المقاطعة إن كانت نافذة منبثقة مفتوحة).

### طابور إعادة الإرسال (pending-sync.json) — منع ضياع فواتير الانقطاع
لو أصدر الكاشير فواتير والخادم متوقف كانت تُحفظ محلياً فقط، وأول بث من المضيف بعد عودة
الاتصال يكتب فوقها فتضيع بصمت. الحل (v1.0.2): كل `postToHost` فاشل يخزّن لقطة الحالة
كاملة في `‎%APPDATA%/sened/pending-sync.json` (محلي لكل جهاز، يصمد أمام إعادة التشغيل،
**خارج البيانات المزامَنة**)، وعند أول اتصال ناجح في `connectClientStream` تُرسَل للمضيف
(`flushPendingSync`) قبل أن يمحوها أي بث وارد — وأي POST ناجح لاحق يمسح الطابور لأنه
أحدث منه (آخر إرسال يفوز). اختبار السيناريو: `node tools/pending-sync-test.js`.

### واجهة الكاشير: بطاقة الربط الأولى وشريط الانقطاع (v1.0.2)
- **بطاقة الربط** (`#lanSetupCard` أعلى صفحة الفواتير): تظهر للكاشير إن كان الجهاز بلا
  إعداد شبكة (role=off) ولم يسبق التخطي — بحث تلقائي + كود اقتران + اختبار + حفظ بدور
  client، بلا مدير وبلا دخول للإعدادات. علم التخطي محلي (`localStorage:
  sened-lan-setup-skipped`) ولا يدخل البيانات المزامَنة. الدوال: `lanSetup*` في `app.js`.
- **شريط الانقطاع** (`#offlineBar` أعلى المحتوى): في وضع العميل عند فقدان الاتصال يظهر
  شريط أحمر لاصق «غير متصل بالخادم — الفواتير تُحفظ محلياً»، النقر يفتح صفحة الشبكة،
  ويختفي عند العودة (`updateOfflineBar` تُستدعى من `updateSyncBadge`).

### التزامن السجلّي (منذ v1.1.0) ونموذج التعارض
التزامن على مستوى **السجل الواحد** بسياسة **آخر كتابة تفوز لكل سجل**: `saveData` يحسب
فروقات المستند سجلاً سجلاً ولا يعبر الشبكةَ إلا ما تغيّر (`POST /sync {changes:[{col,id,data|null}]}`)،
والمضيف يطبّقها ويبثّها (`event: changes` مع `rev`). تعديل جهازين سجلَّين مختلفين في نفس
اللحظة **لم يعُد يتعارض إطلاقاً**؛ التعارض الوحيد المتبقي هو تعديل جهازين **نفس السجل** في
نفس اللحظة (نادر جداً بأدوار متمايزة). **العدّادات لا تتراجع أبداً** (دمج بالأكبر لكل عدّاد في
`applyChangesLocal`) فلا تتكرر أرقام الفواتير. **الشفاء الذاتي:** عند فجوة في أرقام المراجعة
(hello أو قفزة في changes) يسحب العميل الحالة كاملة (`resyncFromHost`). **توافق رجعي:**
المضيف يقبل مستنداً كاملاً من عملاء ما قبل 1.1.0 (يحوّله فروقات)، ويبثّ لهم `event: update`
بالمستند الكامل كما اعتادوا — العملاء الجدد يعلنون `x-sened-proto: 2`.
طابور الانقطاع (`pending-sync.json`) صار قائمة تغييرات مدموجة بالمعرّف بدل لقطة كاملة.

### التحديث التلقائي (منذ v1.1.0)
- **نسخة التثبيت (Setup):** electron-updater مع مزوّد GitHub (المستودع العام — مجاني). يفحص
  بعد الإقلاع بـ30 ثانية ثم كل 4 ساعات، ينزّل بصمت، **ويثبّت عند إغلاق التطبيق** بلا أي نافذة
  (`autoInstallOnAppQuit`). الواجهة تعرض toast فقط (`update:ready` → `T.updateReady`).
- **النسخة المحمولة:** لا تدعم التثبيت الذاتي — تفحص آخر إصدار عبر GitHub API وتعرض تنبيهاً
  (`update:portable` → `T.updatePortable`).
- **CI:** ‏`latest.yml` و`*.blockmap` يُرفعان مع exe إلى صفحة الإصدار (build.yml) — بدونهما
  لا يجد electron-updater التحديث. إعداد `build.publish` في package.json يولّد latest.yml.
- `initAutoUpdate()` لا يعمل في نسخة التطوير (`app.isPackaged`).

## 5. دليل تشغيل الأجهزة الثلاثة على نفس الشبكة

1. **الشبكة:** ضع الأجهزة الثلاثة على نفس شبكة الواي فاي / الشبكة المحلية.
2. **جهاز المدير (الخادم):** الإعدادات → «التزامن عبر الشبكة المحلية» → اختر **الخادم الرئيسي (المضيف)**
   → احفظ. سيظهر عنوان مثل `192.168.1.10:3050` — دوّنه.
   - عند أول تشغيل قد يطلب ويندوز السماح عبر الجدار الناري: **اسمح** (شبكة خاصة/Private).
3. **جهاز المحاسب وجهاز الكاشير (متصلان):** الإعدادات → «التزامن عبر الشبكة المحلية»
   → اختر **جهاز متصل** → أدخل عنوان الخادم (`192.168.1.10`) → اضغط **اختبار الاتصال**
   (يجب أن يظهر «الخادم متاح ✓») → احفظ.
4. **الدخول:** في كل جهاز سجّل الدخول بالدور المناسب (مدير/محاسب/كاشير) فتظهر واجهته وصلاحياته.
5. الآن أي فاتورة أو عملية على أي جهاز تظهر فوراً على البقية.

**ملاحظات:** المنفذ الثابت 3050. لو تغيّر عنوان IP المضيف (إعادة تشغيل الراوتر) حدّث العنوان
في الأجهزة المتصلة — أو احجز IP ثابتاً للمضيف في الراوتر. بلا شبكة يعمل كل جهاز على نسخته
المحلية ويتزامن عند عودة الاتصال (المضيف يبقى المرجع).

## 6. اختبار التزامن السريع

- **تكامل كامل على جهاز واحد (الطريقة المعتمدة منذ v1.1.0):** شغّل نسختين معزولتين عبر
  `SENED_USERDATA` — لكل نسخة مجلد بيانات فيه `lan-config.json` جاهز (host على منفذ تجريبي
  3061 / client على 127.0.0.1). المضيف يكشف HTTP API كاملاً فيُختبر خارجياً: `GET /ping`
  (rev)، `GET /sync` (المستند)، `POST /sync {changes}`، وSSE مع `x-sened-proto: 2`.
  قاعدة العميل تُقرأ مباشرة بـ better-sqlite3 (readonly، آمن مع WAL) للتحقق من وصول التغييرات.
- **ملاحظة:** `tools/sync-test.js` و`tools/pending-sync-test.js` كُتبا لبروتوكول المستند الكامل
  (ما قبل 1.1.0) — المضيف ما يزال يقبله (توافق رجعي) لكنها لا تختبر المسار السجلّي الجديد.
- **فحص نحوي:** `node --check main.js` و`node --check renderer/app.js` قبل أي التزام.
- **better-sqlite3 محلياً (جهاز بلا Visual Studio):** `npm install --ignore-scripts` ثم داخل
  `node_modules/better-sqlite3`:‏ `npx prebuild-install --runtime=electron --target=<إصدار Electron>`
  — يجلب نسخة مسبقة البناء. CI يبني بنفسه (windows-latest فيه أدوات الترجمة).

## 7. طريقة عمل الوكلاء الثلاثة (Agents)

عند العمل على هذا المستودع، وزّع المهام على ثلاثة أدوار — ويُفضَّل تنفيذها **تسلسلياً**
لا بالتوازي، لأن الملفات مشتركة (`main.js`, `app.js`) والتعديل المتوازي يفسدها:

1. **وكيل التنفيذ (Implementation):** يكتب الميزات ويعدّل الكود موافقاً للبنية أعلاه،
   ويضيف مفاتيح الترجمة الثلاثة لأي نص جديد.
2. **وكيل التزامن (Sync):** يصمّم/يعدّل طبقة الشبكة (host/client/SSE/IPC) ويضمن بقاء إعداد
   الشبكة محلياً لكل جهاز خارج البيانات المزامَنة.
3. **وكيل الفحص (QA/Review):** يراجع كل تعديل — الأمان (تسريب مفاتيح)، الصلاحيات، صحّة
   التزامن، ومعرّفات HTML المكررة — ويصدر تقرير تدقيق قبل الدمج.

## 8. تحذيرات مهمة

- **الأرقام غربية دائماً** (`fmt` بـ `en-US`) — لا تُرجعها لأرقام هندية.
- **لا تضع أسراراً في المستودع:** مفاتيح Groq وكود التفعيل ورموز تلجرام تبقى في
  `builtin-ai.json` (gitignored) فقط. أي مفتاح ظهر في المستودع سابقاً يجب **تدويره فوراً**.
- **اختبار main.js** (PDF، النسخ، بوت تلجرام، التزامن الفعلي) يتطلب Electron حقيقياً أو مراجعة
  دقيقة؛ متصفح المعاينة يختبر الواجهة فقط عبر `window.sened` البديل في `app.js`.
- بعد أي `clone` جديد: `npm install` (node_modules مستثنى).
