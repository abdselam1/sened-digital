# خطة: خادم سحابي مركزي (PostgreSQL) متعدّد المؤسسات لـ«سند»

> وثيقة تصميم — كتبها وكيل Claude Code بناءً على طلب المالك (2026-07-14):
> جعل «سند» يعمل عبر الإنترنت لمؤسسات كبيرة، كل مؤسسة لها **كود خاص** وبياناتها معزولة،
> والأجهزة (مدير/محاسب/كاشير) تتزامن تلقائياً عبر خادم PostgreSQL مركزي بدل الشبكة المحلية.

## 1. الفكرة الجوهرية

«سند» 1.1.1 **يملك أصلاً** نظام أدوار (مدير/محاسب/كاشير)، وكود اقتران، وتزامناً سجلّياً
(record-level، آخر كتابة تفوز)، وطابور انقطاع، وقفل دور الجهاز. لكن كل هذا مبني على
**الشبكة المحلية (LAN)**: جهاز «مضيف» يشغّل خادم HTTP على المنفذ 3050، والبقية عملاء.

بدل إعادة كتابة التطبيق، نُبقي منطق «العميل» كما هو ونستبدل **المضيف** بخادم مركزي على
الإنترنت يتكلّم **نفس البروتوكول بالضبط**، لكنه:
- مدعوم بـ **PostgreSQL** (JSONB) بدل ملف SQLite على جهاز.
- **متعدّد المؤسسات**: يعزل بيانات كل مؤسسة حسب كودها.

النتيجة: تغييرٌ صغير جداً في التطبيق (توجيهه لعنوان إنترنت + https)، وخادمٌ جديد مستقل
يحمل كل الثقل.

## 2. تطابق البروتوكول (ما يتوقّعه العميل الحالي)

من `main.js` (دوال `connectClientStream`, `postToHost`, `pullFromHost`):

| نقطة النهاية | الطريقة | ما يرسله/يتوقّعه العميل |
|---|---|---|
| `/ping` | GET | `{ ok, app:'sened', name, rev, clients }` |
| `/sync` | GET | المستند الكامل: `{ settings, counters, <collections…>, _rev, _ts }` |
| `/sync` | POST | جسم `{ changes:[{col,id,data|null}] }` → يعيد `{ ok, rev }` |
| `/events` | GET (SSE) | `event: hello {rev}` ثم `event: changes {rev, changes}` عند كل تحديث |

**الترويسات التي يرسلها العميل مع كل طلب:**
- `x-sened-token` → **نستعمله ككود المؤسسة (tenant)**. لا حاجة لترويسة جديدة.
- `x-sened-device` → اسم الجهاز (للعرض في «الأجهزة المتصلة»).
- `x-sened-proto: 2` → يعني «أرسل لي تغييرات سجلّية لا مستنداً كاملاً».

**سلوك يجب محاكاته على الخادم:**
- **العدّادات لا تتراجع**: عند تطبيق `_kv/counters` خذ الأكبر لكل مفتاح (يمنع تكرار أرقام الفواتير).
- **الحذف شاهدة**: `data:null` = محذوف (لا حذف فيزيائي، كي ينتشر للأجهزة).
- **رقم المراجعة (rev)** يرتفع مع كل تغيير مطبَّق ويُبثّ؛ فجوة > 1 لدى العميل تجعله يسحب كل الحالة.

## 3. مخطّط قاعدة PostgreSQL (متعدّد المؤسسات)

```sql
CREATE TABLE tenants(               -- المؤسسات (كل صف = عميل/مؤسسة بكودها)
  code       TEXT PRIMARY KEY,      -- كود المؤسسة (طويل وعشوائي = مفتاح + سرّ)
  name       TEXT,                  -- اسم المؤسسة (للعرض الإداري)
  rev        BIGINT NOT NULL DEFAULT 0,
  active     BOOLEAN NOT NULL DEFAULT true,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE records(              -- كل السجلات لكل المؤسسات، معزولة بعمود tenant
  tenant     TEXT NOT NULL REFERENCES tenants(code) ON DELETE CASCADE,
  collection TEXT NOT NULL,        -- products/customers/invoices/… أو '_kv'
  id         TEXT NOT NULL,        -- معرّف السجل (أو 'settings'/'counters' لـ _kv)
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  deleted    BOOLEAN NOT NULL DEFAULT false,
  data       JSONB,                -- محتوى السجل (هنا تتفوّق PostgreSQL)
  PRIMARY KEY(tenant, collection, id)
);
CREATE INDEX records_tenant_idx ON records(tenant);
```

**العزل:** كل استعلام يُقيَّد بـ `WHERE tenant = $code`. مؤسسة لا ترى بيانات أخرى إطلاقاً.

## 4. نموذج «الكود» والحسابات (كما طلب المالك)

- **كود المؤسسة** = صف في جدول `tenants`. يولّده الأدمن (أنت) عند اشتراك مؤسسة جديدة،
  ويُسلَّم لمديرها. يجب أن يكون **طويلاً وعشوائياً** (مثلاً 24 محرفاً) لا 6 أرقام، لأنه
  يُستعمل مفتاحاً وسرّاً في آنٍ (من يملكه يدخل بيانات المؤسسة).
- **المدير/المحاسب/الكاشير**: حسابات **داخل بيانات المؤسسة نفسها** (نظام `employees[]`
  و`settings.auth` الموجود أصلاً في التطبيق). كلها تتشارك كود المؤسسة نفسه. يعني:
  المدير يدخل الكود مرة على كل جهاز، ثم يسجّل الدخول بدوره (مدير/محاسب/كاشير) كالمعتاد.
- **التوزيع:** المدير يأخذ الكود منك → يضعه في كل جهاز (شاشة «الاتصال بالسحابة») →
  ينشئ حسابات الكاشير والمحاسب من داخل التطبيق كما يفعل اليوم → الكل يتزامن عبر الإنترنت.

## 5. تغييرات جهة التطبيق (صغيرة — تُنفَّذ وتُختبر لاحقاً مع المالك)

1. **وضع «سحابة» في إعداد الشبكة**: بجانب host/client/off نضيف مفهوم عميل يتصل بعنوان
   إنترنت (نطاق + https) بدل IP محلي + منفذ. عملياً: السماح لـ `hostIp` أن يكون نطاقاً،
   واستخدام وحدة `https` عند المنفذ 443/عنوان يبدأ بـ https.
2. **شاشة إدخال كود المؤسسة**: حقل واحد بسيط يملأ `token` (= الكود) و`hostIp` (= عنوان الخادم،
   قد يكون مضبوطاً مسبقاً في `builtin-ai.json`). أبسط تجربة ممكنة للمستخدم: «الصق الكود واضغط اتصال».
3. **دعم القاعدتين**: الجهاز يبقى SQLite محلياً (offline)، والسحابة هي المزامنة. لا نحذف SQLite.
   العميل الصغير بلا كود = يعمل محلياً فقط (كما اليوم). العميل بكود = يتزامن سحابياً.
4. **https**: دوال `postToHost/connectClientStream/pullFromHost` تستخدم `http` — نضيف فرع `https`.

> ملاحظة: هذه التغييرات في `main.js` تحتاج اختباراً في Electron حقيقي قبل النشر، لذلك
> نتركها لجلسة مع المالك. الخادم أدناه مستقل ويُختبر وحده بـ `node`.

## 6. النشر (خطوات المالك عند الجاهزية)

1. استضافة (VPS مثل Hetzner/DigitalOcean، أو أي خادم Linux) + نطاق + شهادة HTTPS.
2. تثبيت PostgreSQL، إنشاء قاعدة، تشغيل `server/schema.sql`.
3. ضبط `DATABASE_URL` في `server/.env`، تشغيل الخادم خلف HTTPS (Caddy/Nginx).
4. لكل مؤسسة جديدة: `node server/admin.js add "اسم المؤسسة"` → يطبع كود المؤسسة → سلّمه للمدير.
5. في التطبيق: أدخل الكود في شاشة «الاتصال بالسحابة» على كل جهاز.
